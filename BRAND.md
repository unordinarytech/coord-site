# Coord brand language

Coord gives coding agents shared memory.

This guide is the source of truth for product naming, positioning, and written
language across the website. Use it before publishing new marketing copy,
interface text, demos, or documentation.

## Positioning

**Audience:** AI engineering teams using multiple coding agents.

**Category:** Shared memory for coding agents.

**Core promise:** Coord carries useful team context from one agent and task to
the next, so agents do not have to start from scratch.

**One-line description:** Coord gives every coding agent the context your team
has already built.

Coord is not positioned as an agent, an all-in-one workspace, or magic
automation. It is the shared memory layer between the agents a team already
uses.

## Voice

Coord sounds:

- **Precise.** Name the concrete problem, action, or outcome.
- **Confident.** State what is true without hedging or overexplaining.
- **Developer-native.** Use the language technical teams use, not enterprise
  filler.
- **Lightly irreverent.** A dry observation is welcome when it sharpens the
  point.

Coord does not sound:

- grandiose, breathless, or revolutionary
- corporate, abstract, or committee-written
- cute at the expense of clarity
- hostile toward users, competitors, or other tools

## Writing principles

### Lead with the point

Put the useful idea in the first sentence. Headlines should make sense without
supporting copy.

- Use: “Every agent starts with the context your team already has.”
- Avoid: “Unlock a new era of seamless agent collaboration.”

### Prefer concrete language

Name the object and the consequence. Replace broad words such as *everything*,
*powerful*, *smart*, and *seamless* with something a reader can picture.

- Use: “Carry decisions and prior work across agents.”
- Avoid: “Keep everything perfectly in sync.”

### Write like a capable teammate

Use active voice, contractions, and short sentences. Explain the product
without teaching the reader what a coding agent is.

- Use “your team,” “your agents,” and “shared memory.”
- Use “we” only when Coord is genuinely speaking about the team building it.
- Use “you” for a direct action or consequence, not as filler.

### Keep claims honest

Only make claims supported by the current product. Do not imply guaranteed
accuracy, zero setup, complete visibility, or real-time behavior unless those
claims have been verified.

Describe outcomes as what Coord enables, not as an absolute:

- Use: “Agents can pick up the context behind prior work.”
- Avoid: “Your agents will never duplicate work again.”

## Product language

### Preferred terms

| Use | Meaning |
| --- | --- |
| **Coord** | The product name. Capitalize exactly this way. |
| **coding agent** | An AI tool that works on software tasks. Use *agent* after context is established. |
| **shared memory** | Coord’s category and primary capability. |
| **context** | Useful prior work, decisions, and team knowledge available to an agent. |
| **prior work** | Work completed or attempted before the current task. |
| **decision** | A choice whose reasoning may matter to later work. |

### Avoid

- Any former product name or feature name
- “AI teammate,” “second brain,” “hive mind,” or “single source of truth”
- “revolutionary,” “game-changing,” “next-generation,” or “supercharge”
- “seamless,” “effortless,” “magical,” or “just works”
- “talk to each other” when the precise meaning is shared context
- “everything” when the actual information can be named

## Style rules

- Use sentence case for headlines, labels, and calls to action.
- Use contractions when they sound natural.
- Prefer periods over exclamation marks. Use at most one exclamation mark on a
  page.
- Use the serial comma.
- Do not use ampersands in prose.
- Keep headlines under 10 words when the layout allows.
- Keep one idea per sentence.
- Spell out a term before introducing an abbreviation.
- Use numerals for measurements and versions; use words for casual numbers
  from one through nine.
- Do not anthropomorphize Coord. Coord stores, carries, surfaces, or shares
  context; it does not think, understand, or know.

## Humor and profanity

The main narrative stays clean. Humor belongs in examples, terminal scenes, and
small moments where a developer recognizes the problem immediately.

- Prefer dry frustration over forced enthusiasm.
- Mild irreverence is acceptable when it is not aimed at a person or product.
- Do not use profanity in headlines, metadata, calls to action, or core product
  explanations.
- Do not mock users or competing agent tools.
- A joke must still explain the product or the problem. If clarity drops when
  the joke is removed, rewrite the line.

## Message pattern

For a short product story, use this order:

1. **Problem:** coding agents begin without the team’s accumulated context.
2. **Cost:** teams repeat work and lose the reasoning behind decisions.
3. **Product:** Coord provides shared memory across coding agents.
4. **Payoff:** useful context carries from one agent and task to the next.
5. **Action:** invite the reader to test, contribute, or follow the work.

## Examples

| Instead of | Write |
| --- | --- |
| “It helps your agents talk.” | “It gives every agent the same shared memory.” |
| “A shared view of everything happening.” | “Decisions and prior work carry into the next task.” |
| “This is actually incredible.” | “Found the earlier decision. Continuing from there.” |
| “Want to try it?” | “Help us test Coord.” |

## Pre-publish checklist

- [ ] Is the product called **Coord** everywhere?
- [ ] Does the first screen explain a real problem or promise?
- [ ] Is shared memory the primary idea?
- [ ] Can a reader understand each headline without the body copy?
- [ ] Are claims specific, supportable, and free of absolutes?
- [ ] Did we remove hype, filler, and vague words?
- [ ] Is humor limited to places where it adds recognition or meaning?
- [ ] Are capitalization, punctuation, and terminology consistent?
- [ ] Does the call to action say what happens next?
- [ ] Has the copy been checked at desktop and mobile widths?
